# api

NestJS 11 REST + tRPC server. Runs on port 8000.

## Scripts

```bash
bun run dev              # start with file watching (bun runs TS directly)
bun run typecheck        # tsc --noEmit
bun run generate         # generate tRPC types once
bun run generate:watch   # regenerate tRPC types on every router change
bun run test             # unit tests (Jest)
bun run test:e2e         # e2e tests
bun run build            # compile to dist/ for production (nest build)
```

## tRPC ‚Äî Adding a New Router

1. Create `src/<feature>/<feature>.router.ts`:

```ts
import { Injectable } from "@nestjs/common";
import { Input, Query, Router } from "nestjs-trpc";
import { z } from "zod";

@Injectable()
@Router()
export class FeatureRouter {
  constructor(private readonly featureService: FeatureService) {}

  @Query({
    input: z.object({ ... }),
    output: z.object({ ... }),
  })
  myProcedure(@Input() input: { ... }) {
    return this.featureService.doSomething(input);
  }
}
```

2. Add router and service to `providers: []` in the feature module
3. Import feature module in `app.module.ts`
4. Run `bun run generate`

The class name becomes the sub-namespace on the client: `WorkflowsRouter` ‚Üí `trpc.workflowsRouter.*`.

## tRPC ‚Äî Mutations

Use `@Mutation()` instead of `@Query()` for write operations (same signature).

## ‚öÝÔ∏è After changing any procedure signature

Run `bun run generate` (or keep `generate:watch` running) before typechecking `web-ui`.

## Architecture Rules

### Routers are thin ‚Äî always
Routers validate input and delegate. Nothing else.

```ts
// ‚úÖ correct
@Query({ input: getByIdSchema })
getById(@Input() input: { id: string }) {
  return this.workflowsService.findById(input.id);
}

// ‚ùå wrong ‚Äî business logic in router
@Query({ input: getByIdSchema })
async getById(@Input() input: { id: string }) {
  const wf = await this.prisma.workflow.findUnique({ where: { id: input.id } });
  if (!wf) throw new TRPCError({ code: "NOT_FOUND" });
  return wf;
}
```

### Services own all logic
- Business rules live in services, never in routers
- Prisma is only accessed inside services (never in routers or controllers)
- Cross-feature dependencies are injected via NestJS DI (import the module, inject the service)

### Error handling
Always use `TRPCError` with semantic codes:
- `NOT_FOUND` ‚Äî resource doesn't exist
- `BAD_REQUEST` ‚Äî input passes Zod but fails business rules
- `CONFLICT` ‚Äî e.g. triggering a workflow that already has an OPEN event
- `FORBIDDEN` ‚Äî e.g. triggering an inactive workflow

### Type safety
- Never use `any`
- Never use non-null assertions (`!`) without an explanatory comment
- Never cast types to work around TypeScript ‚Äî fix the type

## Business Rules

> These are functional requirements. Breaking them is a bug.

### Workflows
- A workflow can be **active** or **inactive** ‚Äî only active workflows can be triggered
- Trigger type is either `THRESHOLD` or `VARIANCE` and is immutable after creation

### Trigger evaluation (`TriggerService`)
- **Threshold**: compare `actualValue` against `config.value` using `config.operator` (`>`, `<`, `>=`, `<=`, `=`)
- **Variance**: fire if `abs(actualValue - baseValue) / baseValue * 100 > deviationPercent`
- Before creating an event, always check:
  1. Workflow is active ‚Üí else throw `FORBIDDEN`
  2. No existing `OPEN` event for this workflow ‚Üí else throw `CONFLICT`
- Message interpolation: replace `{{metric}}` and `{{value}}` in `outputMessage` before sending notifications
- Manual trigger from the UI must go through the **exact same** `TriggerService.evaluate()` method as an automatic trigger

### Events
- Status is either `OPEN` or `RESOLVED`
- Only one `OPEN` event per workflow at a time ‚Äî enforced in `TriggerService`
- Resolving an event sets `status = RESOLVED` and records `resolvedAt`
- An optional `resolvedComment` can be set only at resolution time
- Once resolved, the workflow can be triggered again

### Notifications
- Notifications are sent **after** the event is created, never before
- Use the interpolated message, not the raw template
- `IN_APP`: store in DB
- `EMAIL`: integrate Nodemailer or log to console (document which in README)
- `NotificationService` must not contain trigger logic ‚Äî it only sends
